## Code review standards

Code review serves two purposes: catching defects before they reach `main`, and spreading knowledge of the codebase across the team. Every PR is reviewed, whether written by a human or generated by an AI agent.

### What reviewers should check

**Always check:**
- Does the code do what the PR description claims?
- Are authorization checks correct (right permission, right layer, super admin bypass present)?
- Are error messages safe (using `ApiErrorMessage` enum, no internal details leaked)? See [backend-error-message-security](backend-error-message-security.md).
- Does the PR include tests at the right layer for the change? See [testing-coverage-expectations](testing-coverage-expectations.md).
- Are Drizzle queries only fetching needed fields?

**Architecture checks:**
- Is business logic in services, not controllers or repositories? See [backend-layer-architecture](backend-layer-architecture.md).
- Is the 5-layer pattern followed?
- Are new dependencies justified, or can existing abstractions be reused?

**Watch for:**
- O(n²) patterns in loops over access control results or embed resolution
- Inline `as any` or `@ts-ignore`. These need justification. See [quality-typescript-strictness](quality-typescript-strictness.md).
- Raw string literals where typed constants exist
- Missing `strictStatusCodes: true` in new API contracts

### AI-generated code: extra scrutiny

AI agents produce correct-looking code that can miss domain-specific edge cases. When reviewing AI-generated PRs, pay extra attention to:

- **Authorization logic:** Agents often get the happy path right but miss edge cases like supervisory vs. supervised role distinctions, or forget the super admin bypass.
- **Access control queries:** Verify the ltree joins are correct and that the right paths (ancestor, descendant, or both) are included.
- **Error handling:** Check that services wrap repository errors and that controllers map the right set of status codes. See [backend-error-handling](backend-error-handling.md).
- **Test assertions:** AI-generated tests sometimes assert on structure without verifying behavior (e.g., checking that a result is an array without checking its contents).
- **Domain semantics:** An agent may use a plausible but wrong field name, enum value, or relationship. Verify against the schema in `apps/backend/src/db/schema/`.

### Self-review before requesting review

Before marking a PR ready for review:

1. Re-read the full diff yourself. Don't rely on having "just written it."
2. Run `npm run lint`, `npm run format:check`, `npm run check-types`, and relevant tests.
3. Verify the PR description accurately reflects the final state of the code.
4. Check that no debugging artifacts remain (console.logs, commented-out code, TODO hacks).

### Incorrect

```
Reviewer: "This variable name could be clearer, but it's fine I guess"
Reviewer: "We usually use early returns here, but this works"
Reviewer: "Approved with minor suggestions"
// PR merged with multiple small issues
```

### Correct

```
Reviewer: "Please rename `d` to `userData` for clarity"
Reviewer: "This should use an early return. See quality-code-style.md"
Reviewer: "The access control join is missing the descendant path for supervisory roles"
// Changes requested. PR updated before merge
```

### The principle

Nits matter. Every pattern violation left in a PR becomes the precedent for the next one. Review is not about being nice or being fast. It's about maintaining the standards that keep an education platform trustworthy. AI-generated code deserves the same (or higher) bar as human-written code, because it can be confidently wrong in ways that are easy to approve without noticing.
